# Design: beneficiaries document duplicates and exchange-rate hard delete rules

## Overview
This change updates business rules in two bounded areas with minimal architectural impact:
- Beneficiaries: remove owner/document uniqueness restriction.
- Exchange rates: hard delete semantics and duplicate-active validation on admin create.

Existing module structure remains unchanged:
- resolver -> use-case -> ports -> Prisma adapters.

## Prisma changes

### Beneficiary constraint change
Current model contains:
- `@@unique([ownerUserId, documentNumber])`

Design change:
1. Remove the unique constraint on `(ownerUserId, documentNumber)`.
2. Preserve lookup performance by adding/keeping a non-unique index:
   - `@@index([ownerUserId, documentNumber])`
3. Keep existing indexes (`ownerUserId`, `ownerUserId+isDeleted`, `ownerUserId+isFavorite`).

No exchange-rate schema changes are required for this rule set.

## Exchange-rate behavior changes

### adminDeleteExchangeRate -> hard delete
Current behavior:
- `deleteExchangeRate` in command adapter updates `enabled=false`.

New behavior:
- `deleteExchangeRate` performs physical delete (`prisma.exchangeRate.delete`).

Expected impact:
- `adminDeleteExchangeRate` mutation contract remains `Boolean`.
- Existing read/list/latest queries remain unchanged and naturally stop returning deleted rows.

### adminCreateExchangeRate duplicate-active validation
Current behavior:
- validates currencies/rate, then always creates.

New behavior:
- Before creation, if requested new row is active (`enabled=true`, default true), validate that there is no existing active row for same normalized pair (`from`, `to`).
- If an active duplicate exists, throw `ValidationDomainException`.
- If only disabled historical rows exist, creation is allowed.

## Validation design

### Duplicate-active definition
A duplicate-active exchange rate means:
- same normalized currency pair (`fromCode`, `toCode`) after trim + uppercase normalization,
- existing row with `enabled=true`.

Validation rule:
- Trigger duplicate check only when incoming create request is active (`input.enabled ?? true` equals `true`).
- Disabled creates (`enabled=false`) do not require active-uniqueness enforcement.

### Beneficiary duplicate document behavior
- No application-layer duplicate rejection added.
- Create/update keep current ownership and not-found rules only.
- Duplicate `documentNumber` per owner is allowed by persistence and business behavior.

## Impacted components

### Beneficiaries
- Prisma schema:
  - `src/prisma/schema.prisma`
- Migration:
  - drop unique beneficiary owner/document constraint
  - add non-unique index if needed
- Beneficiary use-cases, ports, resolvers, adapters:
  - no behavior change expected unless any hidden uniqueness assumption appears during verification.

### Exchange rates
- Use-cases:
  - `src/modules/exchange-rates/application/use-cases/admin-create-exchange-rate.usecase.ts`
  - `src/modules/exchange-rates/application/use-cases/admin-delete-exchange-rate.usecase.ts` (signature unchanged)
- Ports:
  - `ExchangeRatesQueryPort` reused for active duplicate lookup (no contract change required)
  - `ExchangeRatesCommandPort` signature unchanged
- Adapters:
  - `src/modules/exchange-rates/infrastructure/adapters/prisma-exchange-rates-command.adapter.ts`
    - delete path becomes physical delete
- Resolver:
  - `src/modules/exchange-rates/presentation/graphql/resolvers/exchange-rates.resolver.ts`
  - no GraphQL shape changes

## Migration considerations
- Generate Prisma migration for beneficiary constraint change.
- Migration should safely drop the unique constraint/index generated by Prisma for `(ownerUserId, documentNumber)`.
- No data backfill is needed.
- Hard delete does not require schema migration.

## Out of scope
- No auth/guards changes.
- No role or RBAC modifications.
- No exchange-rate query contract redesign.
- No new dedup logic for beneficiaries beyond removing uniqueness restriction.
- No broad refactors outside the two rule areas.
